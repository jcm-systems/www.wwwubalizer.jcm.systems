WWWubalizer

What's this then?

The WWWubalizer sends people who manually type in your apex (or "naked") domain over to your main website.

In other words, someone who types in http://your-domain.com will end up at http://www.your-domain.com

How can I use it?

Just place a DNS "A" record at the apex of your domain, pointing at this IP address: 35.178.28.1

That's it? I don't have to ask you to set it up for me, or anything?

Yep. That's it.

Why?

Because it's annoying to have to set up something to do this, with TLS, for each and every new site you launch!

The WWWubalizer is a domain-agnostic, permanent, static-IP-based service ...

... which works even when your more security-minded visitors type in https://your-domain.com.

What does the service actually do?

A webserver:

  • listens for requests, and ...

  • responds with an HTTP 301 permanent redirect ...

  • with the Location header set to the root of your www site ...

  • using the same scheme (HTTP/HTTPS) that the request used originally.

Also, a valid TLS certificate is obtained, served and cached when an HTTPS request is received.

What DOESN'T the service do?

This service, under non-abuse conditions, is guaranteed NOT to:

  • Serve anything other than an HTTP 301 redirect to "http(s)://www.{yourdomain}/"

In all conditions, this service is guaranteed NOT to:

  • Insert any response headers which will harm your domain's visitors

  • Set any cookies

  • Set any security related headers such as Content-Security-Policy or Strict-Transport-Security

  • Serve, use, or expose any Javascript or other code which your domain's visitors' browsers might try to execute

  • Use any TLS certificates obtained for evil. Or indeed for any purpose other than serving 301 redirects as described above

Data collection

All this data is used solely to help run the service smoothly. None of it is being shared anywhere, and none of it will be exploited in any way at all. Ever. Here's what's collected:

  • The time the request was made

  • The first 3 octets of the IPv4 address making the request

  • The domain being requested

  • The scheme the request used

  • The HTTP method the request used

  • The HTTP referer

Why are you collecting the HTTP referer?

The point of this service is to provide an easy way to deal with your domain's users who type in your domain, and not those who simply click on a link to your apex domain. Whilst I'm not blocking (and not planning to block) any requests which contain a referer (which would indicate you've put a link to your naked domain somewhere, intentionally, thereby making this service part of your users' experience), I need to collect this information to figure out if this is a reasonable state of affairs. Long story short: where you control the HTML, link to your actual website ("www.your-domain.com"), not your apex domain!

Is this service compliant with the GDPR?

This service:

  • isn't operated as part of any "economic activity",

  • doesn't store any personally identifiable IP addresses,

  • doesn't store the HTTP path or query string that any user clicked on, or typed in.

It's my understanding that this both puts the service out of scope of the GDPR, and adheres to its rules in case I'm mistaken and it turns out it is in scope.
